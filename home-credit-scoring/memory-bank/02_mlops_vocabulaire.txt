                            1. TERMES DE SCORING CREDIT
================================================================================

SCORING CREDIT
  Definition: Methode d'evaluation automatisee du risque de defaut d'un
              emprunteur basee sur des modeles statistiques ou ML.
  Avantages: Objectivite, rapidite, standardisation des decisions
  Inconvenients: Peut manquer de nuance, necessite des donnees de qualite

TARGET (Variable cible)
  Definition: Variable binaire a predire
    - 0 = Client qui rembourse normalement (non-defaut)
    - 1 = Client qui fait defaut (ne rembourse pas)
  Dans ce projet: TARGET presente dans application_train.csv

PROBABILITE DE DEFAUT
  Definition: Score entre 0 et 1 representant la probabilite qu'un client
              fasse defaut sur son credit.
  Utilisation: Comparee a un seuil pour la decision finale

SEUIL DE DECISION (Threshold)
  Definition: Valeur de probabilite au-dela de laquelle un client est
              considere comme a risque (prediction = 1)
  Par defaut: 0.5
  Optimal (ce projet): 0.44 (base sur le cout metier)

FAUX NEGATIF (FN - Type II Error)
  Definition: Client en defaut (1) predit comme non-defaut (0)
  Consequence: Credit accorde a un mauvais payeur -> perte financiere
  Cout dans ce projet: 10

FAUX POSITIF (FP - Type I Error)
  Definition: Bon client (0) predit comme defaut (1)
  Consequence: Credit refuse a un bon client -> manque a gagner
  Cout dans ce projet: 1

COUT METIER
  Definition: Fonction de cout personnalisee integrant les consequences
              financieres des erreurs de prediction
  Formule: Cout = (FN * 10) + (FP * 1)
  Objectif: Minimiser ce cout plutot que maximiser l'accuracy

AUC-ROC (Area Under the Receiver Operating Characteristic Curve)
  Definition: Mesure de la capacite du modele a distinguer les classes
  Plage: 0.5 (aleatoire) a 1.0 (parfait)
  Ce projet: AUC = 0.7683
  Attention: AUC > 0.82 suggere de l'overfitting sur ce dataset

PRECISION
  Definition: Proportion de vrais positifs parmi les predictions positives
  Formule: TP / (TP + FP)
  Ce projet: 0.1724 avec seuil optimal

RECALL (Sensibilite)
  Definition: Proportion de vrais positifs parmi les positifs reels
  Formule: TP / (TP + FN)
  Ce projet: 0.7026 avec seuil optimal
  Important: Recall eleve = moins de FN = moins de pertes

================================================================================
                            2. TERMES MLOPS
================================================================================

MLOPS
  Definition: Ensemble de pratiques pour deployer et maintenir des modeles
              ML en production de maniere fiable et efficace.
  Composants: CI/CD, tracking, monitoring, versioning

MLFLOW
  Definition: Plateforme open source pour le cycle de vie ML
  Composants utilises:
    - Tracking: Log des parametres, metriques, artifacts
    - Model Registry: Stockage centralise des modeles
    - UI: Interface web pour visualiser les experiences

TRACKING (MLflow)
  Definition: Enregistrement automatique des:
    - Parametres (hyperparametres du modele)
    - Metriques (AUC, accuracy, cout metier)
    - Artifacts (graphiques, fichiers CSV, modeles)
  Avantage: Reproductibilite, comparaison des experiences
  Implementation: mlflow.log_param(), mlflow.log_metric(), mlflow.log_artifact()

MODEL REGISTRY (MLflow)
  Definition: Registre centralise pour stocker et versionner les modeles
  Fonctionnalites:
    - Versioning des modeles
    - Stages (Staging, Production, Archived)
    - Lineage (tracabilite)
  Implementation: mlflow.register_model()

RUN (MLflow)
  Definition: Une execution unique d'entrainement avec ses parametres,
              metriques et artifacts associes.
  Identifiant: run_id unique

EXPERIMENT (MLflow)
  Definition: Groupe de runs lies a un meme objectif
  Ce projet: Experiment "home-credit-scoring"

DATA DRIFT
  Definition: Changement dans la distribution des donnees d'entree
              entre l'entrainement et la production.
  Detection: Comparaison statistique train vs nouvelles donnees
  Outil: Evidently (bibliotheque Python)

EVIDENTLY
  Definition: Bibliotheque Python pour le monitoring ML
  Fonctionnalites:
    - Data Drift detection
    - Data Quality reports
    - Model Performance monitoring
  Output: Rapports HTML interactifs

CI/CD (Continuous Integration / Continuous Deployment)
  CI: Integration automatique du code (tests, linting)
  CD: Deploiement automatique en production
  Outil: GitHub Actions
  Ce projet: ci.yml (tests) + deploy.yml (deploiement)

================================================================================
                            3. TERMES TECHNIQUES ML
================================================================================

LIGHTGBM (Light Gradient Boosting Machine)
  Definition: Algorithme de gradient boosting sur arbres de decision
  Avantages:
    - Tres rapide (optimisation histogramme)
    - Bonne performance sur donnees tabulaires
    - Gere les valeurs manquantes nativement
    - Supporte class_weight pour desequilibre
  Inconvenients:
    - Boite noire (moins interpretable)
    - Peut overfitter si mal parametre

GRADIENT BOOSTING
  Definition: Technique d'ensemble qui combine des modeles faibles
              (arbres) sequentiellement pour reduire l'erreur.
  Principe: Chaque arbre corrige les erreurs du precedent

CLASS_WEIGHT='BALANCED'
  Definition: Pondere les classes inversement a leur frequence
  Effet: Donne plus d'importance aux classes minoritaires
  Alternative: SMOTE, undersampling, oversampling

EARLY STOPPING
  Definition: Arret de l'entrainement quand la performance sur
              validation ne s'ameliore plus.
  Ce projet: 50 rounds sans amelioration -> arret

CROSS-VALIDATION (Validation Croisee)
  Definition: Division des donnees en K folds pour evaluer
              la robustesse du modele.
  Ce projet: StratifiedKFold (preserve les proportions de classes)

FEATURE IMPORTANCE
  Definition: Mesure de la contribution de chaque variable au modele
  Types:
    - Globale: Impact moyen sur tout le dataset
    - Locale: Impact sur une prediction specifique (SHAP)
  Top features ce projet: EXT_SOURCE_2, EXT_SOURCE_3, DAYS_BIRTH

SHAP (SHapley Additive exPlanations)
  Definition: Methode d'interpretabilite basee sur la theorie des jeux
  Principe: Calcule la contribution de chaque feature a une prediction
  Avantage: Explications locales et globales coherentes
  Usage: Expliquer au client pourquoi son credit est refuse/accepte

PREPROCESSING
  Definition: Preparation des donnees avant entrainement
  Etapes typiques:
    - Nettoyage (valeurs manquantes, outliers)
    - Encodage (categoriel -> numerique)
    - Normalisation/Standardisation
    - Feature engineering

FEATURE ENGINEERING
  Definition: Creation de nouvelles variables a partir des existantes
  Exemples ce projet:
    - CREDIT_INCOME_RATIO = AMT_CREDIT / AMT_INCOME_TOTAL
    - AGE_YEARS = -DAYS_BIRTH / 365
    - EXT_SOURCE_MEAN = moyenne des scores externes

FASTAPI
  Definition: Framework Python moderne pour creer des APIs REST

    - Tres performant (async)
    - Documentation automatique (Swagger/OpenAPI)
    - Validation automatique (Pydantic)

ENDPOINT
  Definition: URL specifique exposant une fonctionnalite de l'API
  Exemples: /predict, /health, /model/info

DOCKER
  Definition: Technologie de conteneurisation
  Avantage: Environnement reproductible, deploiement simplifie
  Ce projet: Dockerfile pour l'API

RENDER
  Definition: Plateforme cloud pour deployer des applications
  Avantages:
    - Tier gratuit disponible
    - Deploiement depuis GitHub
    - Support Docker
    - HTTPS automatique

STREAMLIT
  Definition: Framework Python pour creer des dashboards web
  Avantages:
    - Simple a utiliser
    - Interactif
    - Deploiement facile (Streamlit Cloud)
